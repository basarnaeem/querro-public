# Evaluation

## What this document is

This document describes how Querro reasons about reliability, output quality, and failure modes for the AI components of the system. It is a methodology document, not a published benchmark. The platform is in active development and a production-scale evaluation harness is being built alongside the v2.0 feature set; this document explains the principles that harness implements and the failure taxonomy it is designed to catch.

The reason this document exists at this stage of the company is straightforward. In a deal advisory product, the cost of an undetected output error is paid by the user's client, not by the user. That asymmetry forces the evaluation question to the front of the design process rather than to the end of it.

## What is and is not in scope for evaluation

The scope of evaluation is bounded by the [AI System Design](03-ai-system-design.md). Querro uses AI in a small number of well-defined places. Evaluation focuses on those places.

In scope for ongoing measurement:

- Adaptive question suggestions — relevance, novelty, dismissal rate
- Briefing memo executive summary — factual grounding, coverage, tone calibration
- Briefing memo consolidated risk section — coverage of flagged risks, hallucination of unflagged risks, narrative coherence
- Document tagging and pre-fill — citation accuracy, extraction accuracy, false confidence on missing data

Out of scope, by design:

- The composition of the questionnaire. The composition is deterministic and reproducible. Evaluation here is a unit-test problem, not a stochastic-output problem. Given identical inputs and library version, the output is identical and verified by automated tests.
- The body of the briefing memo. Sections of the memo other than the executive summary and risk narrative are rendered from structured fields by a deterministic compiler. There are no model outputs to evaluate.
- Financial model values. Model values are computed by spreadsheet templates from user-entered assumptions. There is no model output to evaluate.

This scope discipline is itself an evaluation choice. A great deal of effort in evaluating AI products is spent measuring outputs that need not have been generated by AI in the first place. Querro avoids this by structuring the system so that fewer outputs are AI-generated.

## The error taxonomy

Every AI feature in Querro is reviewed against five categories of failure. These categories were chosen because each maps to a different mitigation strategy and a different observability signal.

### Category 1 — Factual fabrication

The model produces a claim, number, or entity that does not appear in the inputs and does not exist in the world. This is the highest-severity category. A fabricated client revenue figure, a fabricated regulator name, or a fabricated risk that does not correspond to a flagged structured field is unacceptable in a deal artifact.

**Mitigation:** Every AI feature in production restricts the model's input to bounded structured fields. The model cannot invent a number that is not in its input. Where free-text input is necessary (the document pre-fill case), every output is tagged with a citation that must resolve to a real source.

**Observability:** For the briefing memo's executive summary and risk section, every numeric or named entity in the AI output is checked against the source structured fields by an automated post-generation validator. Outputs that contain unsourced numbers or unsourced entity names are rejected and regenerated. This validator is in production today.

### Category 2 — Citation drift

The model produces a claim that is technically supported by the input but cites the wrong source — for example, attributing a customer concentration risk to the operational section when it was flagged in the financial section.

**Mitigation:** Each input field carries a stable identifier. The model is instructed to produce outputs in a structured format that names the source identifier for each claim. A post-generation step validates that the named source identifier actually contains the claimed information.

**Observability:** Same automated validator as Category 1. Citation mismatches are caught before the output reaches the user.

### Category 3 — Coverage failure

The model omits a claim that should have been included — for example, a flagged risk that does not appear in the consolidated risk section.

**Mitigation:** The set of flagged risks is computed deterministically from the structured engagement record. The model is given the full set as input. A post-generation validator confirms that every input risk appears in the output narrative; missing risks trigger regeneration with the missing items emphasized.

**Observability:** Same validator. This category is measured by counting the ratio of input flagged risks to mentioned risks in the generated narrative.

### Category 4 — Tone and register failure

The model produces output that is factually correct but stylistically wrong for a deal document — too casual, too verbose, too marketing-driven, or formatted inconsistently with the rest of the memo.

**Mitigation:** Output style is controlled by a fixed system instruction defining the voice (terse, factual, banker-register) and by examples drawn from a hand-curated set of approved memo sections. Style violations are caught in user review and recorded as feedback.

**Observability:** This category is harder to automate. Today it is measured by user feedback on the regenerate button — if a user regenerates the same section more than once, the regeneration cause is logged and reviewed.

### Category 5 — Confidence calibration failure

The model produces an output that asserts something with implicit confidence that the underlying data does not support — for example, summarizing an answer that the user marked as "needs verification" without surfacing that uncertainty.

**Mitigation:** Confidence-bearing fields in the engagement record (verification flags, sensitivity flags, "to be confirmed" markers) are passed to the model as explicit signals, and the model's output template requires their preservation. A summary of an unverified field must mark it as unverified.

**Observability:** Automated validator confirms that every flagged-as-unverified input field whose content appears in the output is also marked as unverified in the output. This is the most recent addition to the validator and is being refined.

## Methods, in order of investment

Three evaluation methods are in use, weighted by how much engineering investment each one has received.

### Method A — Manual review on every AI output (in production today)

Every output of every AI feature is reviewed by the user before it commits to the engagement record. For the briefing memo, this is the user clicking through the generated document and accepting, regenerating, or rewriting each AI-touched section. For document pre-fill, it is the user clicking through each pre-filled answer with its source citation visible.

This is the most important evaluation method in the current product, not because it is sophisticated but because it is universal. No AI output enters the engagement record without a human review event. The system is designed so that this review is fast — a few seconds per AI-touched section — and so that the review is forced by the workflow, not optional.

### Method B — Automated post-generation validation (partially in production)

Described in the error taxonomy above. The validator runs after every AI generation, checks the structured properties (sourced numbers, cited identifiers, coverage of flagged inputs, preservation of uncertainty markers), and either passes the output through to the user or triggers a constrained regeneration.

The validator is fully in production for Categories 1 and 2 (factual fabrication and citation drift). Category 3 (coverage) is in production for the consolidated risk section. Categories 4 and 5 are partially instrumented and being expanded.

### Method C — Held-out engagement evaluation set (in design)

A set of historical engagements — the founder's own deal experience plus contributions from advisors and pilot customers — is being curated into a versioned evaluation set. Each engagement in the set has a hand-graded "correct" briefing memo and risk register. Changes to the AI prompts, the model version, or the composition library are tested against this set before being deployed.

This method is not yet running on a continuous integration cadence. It is the next investment after the v2.0 feature set ships. Constructing an evaluation set that is representative across mandate type, business model, sub-industry, jurisdiction, and special situation is a non-trivial sampling problem and is being scoped carefully.

## What good looks like

The project's near-term targets, applied per AI feature, are these.

| Metric | Target | Current state |
|--------|--------|---------------|
| Factual fabrication rate (Category 1) | < 0.5% of generated sections | Validator catches and regenerates; observed pass-through rate to user is below target. Sample size remains modest pending pilot rollout. |
| Citation drift rate (Category 2) | < 1% of cited claims | Validator catches and regenerates; same caveat. |
| Coverage failure rate (Category 3) | < 2% of flagged inputs missed | Currently measured for the risk section only; coverage on executive summary is in scope for the next iteration. |
| User regeneration rate (Category 4 proxy) | < 15% of generated sections | Logged but not yet a primary KPI. |
| Calibration preservation (Category 5) | 100% of flagged-as-unverified inputs preserved | Validator under refinement; current performance imperfect on edge cases. |

These targets are deliberately set at production-grade thresholds rather than at research-grade thresholds. The product needs to be reliable for paying customers running real deals, not to win on a public leaderboard.

## What this document does not promise

Three things are explicitly not claimed.

**This is not a published benchmark.** The metrics above are internal targets and partial-state measurements, not a public score on a held-out test set. The platform's pilot rollout, expected in the third quarter of 2026, will be the first meaningful production volume against which to compute publishable numbers.

**This is not a guarantee.** Every AI system fails sometimes. The architecture is built so that those failures are bounded, detectable, and reviewable. Failures will still occur, will still be documented, and will continue to drive iteration.

**This is not the complete picture.** The internal evaluation tooling, the prompt-versioning infrastructure, the regression-test harness, and the curated evaluation engagements are proprietary. This document describes the methodology and the observed properties; it does not describe the implementation.
